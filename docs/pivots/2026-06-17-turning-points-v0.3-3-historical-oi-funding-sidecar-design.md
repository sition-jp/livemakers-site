# LiveMakersCom AI Turning Point Detector v0.3-3 Historical OI/Funding Sidecar Design

> Date: 2026-06-17
> Status: Design for review
> Product: LiveMakersCom AI Turning Point Detector
> Scope: v0.3 third slice, historical derivatives data layer
> Positioning: SITION Market State Intelligence Engine leverage-history foundation

## Context

v0.3-1 added current-snapshot derivatives evidence from Binance REST inputs:
Open Interest, funding, global long/short ratio, and top-trader position ratio.
v0.3-2 settled the methodological follow-ups from the v0.3-1 review: last
closed ratio buckets, current-excluded percentile baselines, top-trader
divergence `raw_value`, and scorer intent comments.

The remaining structural gap is historical leverage data. Current production
can explain today's derivatives conditions, but the backtest still relies on
price-derived proxies for historical OI and stubbed funding. That is acceptable
for a hidden beta and evidence-only display, but it is not enough for:

- promoting derivatives evidence into score math;
- making the backtest tunable against actual leverage/funding regimes;
- consuming Turning Point output inside AI Auto Trader;
- treating confidence quality as trade-relevant.

The previous gate remains in force:

```text
AI Auto Trader connection waits until a historical OI/funding sidecar exists
and the backtest can use it as real historical evidence.
```

Provider constraints matter:

- Binance USD-M `openInterestHist` supports `period`, `limit`, `startTime`, and
  `endTime`, but the official docs state only the latest 1 month is available.
- Binance USD-M funding history supports `startTime`, `endTime`, and max
  `limit=1000`; it returns records in ascending order.
- Binance long/short ratio and top-trader position ratio endpoints also expose
  only the latest 30 days.
- Actual liquidation data remains stream-oriented and is still out of scope.

References:

- Binance Open Interest Statistics:
  `https://developers.binance.com/docs/derivatives/usds-margined-futures/market-data/rest-api/Open-Interest-Statistics`
- Binance Funding Rate History:
  `https://developers.binance.com/docs/derivatives/usds-margined-futures/market-data/rest-api/Get-Funding-Rate-History`
- Binance Long/Short Ratio:
  `https://developers.binance.com/docs/derivatives/usds-margined-futures/market-data/rest-api/Long-Short-Ratio`
- Binance Top Trader Long/Short Position Ratio:
  `https://developers.binance.com/docs/derivatives/usds-margined-futures/market-data/rest-api/Top-Trader-Long-Short-Ratio`

## Goal

Create a durable sidecar data layer that accumulates daily BTC/ETH Open
Interest and funding history from public Binance REST endpoints.

The sidecar should answer:

```text
What did leverage and funding conditions look like on each closed UTC day,
and can the producer/backtest safely use that history later without relying
on today's short provider lookback window?
```

## Non-Goals

- No score change in this slice.
- No confidence-grade change in this slice.
- No public schema version bump for `pivot_assets.live.json`.
- No public schema version bump for `pivot_backtest.live.json`.
- No UI, nav, API route, or zod contract change for public Turning Point pages.
- No AI Auto Trader connection.
- No SDE ingestion path.
- No WebSocket liquidation collector.
- No paid provider or provider key.
- No database service.
- No separate LaunchAgent.
- No historical backfill claim beyond what the source endpoints actually expose.

## Considered Approaches

### A. Integrated Daily OI/Funding Sidecar

Add a third repo data artifact generated by the existing daily producer path:

```text
data/pivot_derivatives_history.live.json
```

The artifact stores normalized daily OI and funding history by asset. Existing
`pivot_assets.live.json` and `pivot_backtest.live.json` remain unchanged.

Pros:

- Starts accumulating the missing history immediately.
- Reuses the existing LaunchAgent, lock, logging, archive, prune, and
  auto-commit operational path.
- No new secret or daemon.
- Keeps the public product contract stable.
- Creates a clean future input for backtest calibration.

Cons:

- Requires upgrading the producer promotion path from two files to three files.
- Requires updating run_daily auto-commit scope.
- The first usable OI history is limited to about 30 closed days plus forward
  accumulation.

### B. Separate Sidecar CLI And Manual/Scheduled Run

Add a standalone `producer.run_derivatives_sidecar` CLI and run it manually or
with a second scheduler.

Pros:

- Smaller change to the existing producer promotion logic.
- Failure is fully isolated from public snapshot generation.

Cons:

- Easy to forget or drift from the daily producer.
- Adds ops surface and another schedule to monitor.
- Slower path to a reliable historical cache.

### C. External Historical Derivatives Provider

Use a paid or external provider with deeper historical OI/funding/liquidation
coverage.

Pros:

- Could unlock multi-year backtests faster.
- May include true liquidation history.

Cons:

- Adds cost, key management, provider lock-in, terms review, and new failure
  modes.
- Premature before the local sidecar contract is stable.

## Decision

Use Approach A.

This slice is named:

```text
turning-points-v0.3-3-historical-oi-funding-sidecar
```

It is a historical data foundation, not a scoring-model release. The sidecar is
allowed to be committed by the daily ops path, but public snapshots must remain
contract-stable.

## Sidecar Artifact

### Path

Canonical live path:

```text
data/pivot_derivatives_history.live.json
```

Archive path:

```text
data/pivots-history/pivot_derivatives_history.live.<YYYYMMDDTHHMMSSZ>.json
```

Rationale:

- The file belongs next to `pivot_assets.live.json` and
  `pivot_backtest.live.json`.
- Existing ops archive/prune naming can extend naturally.
- The artifact is repo-visible and reviewable, but not public API surface.

### Schema

Initial schema version:

```text
pivots_derivatives_history.v0.1
```

Shape:

```json
{

  "generated_at": "2026-06-17T23:00:18Z",
  "provider": "binance_usdm",
  "assets": {
    "BTC": {
      "symbol": "BTCUSDT",
      "history": [
        {
          "bucket_start": "2026-06-16T00:00:00Z",

          "open_interest": {
            "sample_count": 6,
            "first": 67432.1,
            "last": 68123.4,
            "min": 67001.2,
            "max": 69002.5,
            "avg": 68010.3,
            "last_usd": 7123456789.0,
            "avg_usd": 7012345678.0,
            "growth_pct": 0.0103
          },
          "funding": {
            "sample_count": 3,
            "sum": 0.0003,
            "avg": 0.0001,
            "abs_avg": 0.0001,
            "max_abs": 0.0002,
            "last": 0.0001
          },
          "completeness": {
            "open_interest": 1.0,
            "funding": 1.0,
            "overall": 1.0
          },
          "source": {
            "oi_period": "4h",
            "funding_granularity": "8h",

          }
        }
      ]

    }

```

The implementation may omit `min`/`max` or USD aggregates if discovery shows the
fixture and review value is not worth the payload size, but `last`,
`growth_pct`, funding `avg`, funding `abs_avg`, and `completeness` should remain.

### Closed-Bucket Rule

The sidecar must only store closed UTC-day buckets.

At the normal 08:00 JST run, the wall clock is 23:00 UTC on the previous local
day. The current UTC day is not closed. Therefore the latest eligible bucket is:

```text
UTC date strictly before generated_at.date()
```

Example:

```text
generated_at = 2026-06-17T23:00:18Z
latest stored bucket = 2026-06-16T00:00:00Z -> 2026-06-17T00:00:00Z
```

This is intentionally conservative. One-day latency is better than quietly
mixing partial OI/funding into a historical cache.

## Data Normalization

### Open Interest

Input:

```text
GET /futures/data/openInterestHist?symbol=<SYMBOL>&period=4h&limit=180
```

Default collection:

- `period=4h`
- `limit=180`
- expected closed-day samples: 6
- source retention: latest 1 month

Daily aggregation:

- group by UTC day using `timestamp`;
- discard current UTC day;
- store only days with at least 1 sample;
- compute completeness as `min(sample_count / 6, 1.0)`;
- compute `growth_pct` as `(last - first) / first` when `first > 0`, else 0.

### Funding

Input:

```text
GET /fapi/v1/fundingRate?symbol=<SYMBOL>&limit=1000
```

Default collection:

- expected funding samples per UTC day: 3
- funding times are event timestamps, not continuous bars;
- deeper pagination may be added later using `startTime`/`endTime`, but v0.3-3
  should not depend on long historical provider reach.

Daily aggregation:

- group by UTC day using `fundingTime`;
- discard current UTC day;
- store only days with at least 1 sample;
- compute completeness as `min(sample_count / 3, 1.0)`;
- store signed `sum`, signed `avg`, `abs_avg`, `max_abs`, and `last`.

### Existing Sidecar Merge

The sidecar update is an upsert, not an append-only blind write:

1. Read existing `data/pivot_derivatives_history.live.json` if present.
2. Fetch latest provider windows for each asset.
3. Aggregate closed UTC days.
4. Upsert by `(asset, bucket_start)`.
5. Sort ascending by `bucket_start`.
6. Prune to the configured retention window.
7. Write a new full sidecar snapshot.

Default retention:

```text
1825 closed days per asset
```

This target mirrors a five-year backtest horizon, but the file must not claim
five years of OI history until enough history has actually accumulated or an
approved historical provider is added.

## Producer Integration

### Preferred Module Boundary

Create a focused sidecar module:

```text
scripts/pivots/producer/derivatives_sidecar.py
```

Responsibilities:

- define sidecar TypedDict/dataclass structures;
- load and validate existing sidecar JSON;
- aggregate OI/funding rows into closed UTC-day points;
- merge/upsert/prune history;
- return a complete sidecar payload;
- expose small query helpers for future backtest work.

Suggested top-level functions:

```python
compose_derivatives_history_sidecar(
    fetcher: BinanceFetcher,
    generated_at: str,
    existing: DerivativesHistorySnapshot | None = None,
    retention_days: int = 1825,
) -> DerivativesHistorySnapshot

load_derivatives_history_sidecar(path: Path) -> DerivativesHistorySnapshot | None
```

### Producer CLI

Extend `producer.run_producer` to accept:

```text
--derivatives-history-path data/pivot_derivatives_history.live.json
```

The default live run should produce three artifacts:

```text
data/pivot_assets.live.json
data/pivot_backtest.live.json
data/pivot_derivatives_history.live.json
```

Dry-run should write and validate a temporary sidecar file, then remove it just
like the existing two snapshots.

### Atomic Promotion

Upgrade promotion from a hard-coded two-file pair to a small N-file promotion
helper.

Required invariant:

```text
If promotion fails after any target is replaced, all replaced targets are rolled
back or the run exits with preserved .bak files and a loud failure.
```

The implementation should keep existing `.bak` refusal behavior and include the
sidecar in the orphan-bak check.

### Zod/Public Validation

Do not add sidecar to the public zod snapshot validator in this slice unless a
reviewer explicitly asks for it.

Instead:

- keep existing zod validation for the public assets/backtest snapshots;
- add Python unit tests for sidecar schema and aggregation behavior;
- add a Python shape validator in `derivatives_sidecar.py` if useful.

Reason:

- The sidecar is not a public API contract yet.
- We want the public schema to remain quiet while the internal data layer
  stabilizes.

## Ops Integration

### run_daily

Extend `scripts/pivots/ops/run_daily.py` to treat the sidecar as a third target:

```text
assets_path
backtest_path
derivatives_history_path
```

The daily sequence remains:

```text
dry-run
-> live write
-> archive + prune
-> scoped auto-commit
-> OK/FAILED log + optional Telegram heartbeat
```

### Auto-Commit Scope

The `git commit --only` protection must remain. The target path list should
expand from two paths to three explicit paths:

```text
data/pivot_assets.live.json
data/pivot_backtest.live.json
data/pivot_derivatives_history.live.json
```

No `git add .`.

### Sidecar Failure Policy

During v0.3-3, sidecar fetch/aggregation failure should not block the public
assets/backtest snapshots.

Recommended behavior:

- If public assets/backtest composition fails: run status is `FAILED`.
- If sidecar update fails but public assets/backtest succeed:
  - preserve the previous sidecar;
  - write an `OK` ops log with warning text in `details`;
  - send the normal OK heartbeat if `notify_ok` is enabled;
  - do not emit partial sidecar data.

Rationale:

- Public hidden-beta continuity matters.
- The sidecar is a new historical cache. Missing one day is repairable; failing
  the daily public snapshot because an optional history layer had a transient
  provider issue is too brittle.

If reviewers prefer stricter durability, v0.3-4 can add a dedicated `WARN`
status to `ops.alert.AlertPayload`. v0.3-3 should avoid widening the alert
contract unless necessary.

## Backtest Policy

v0.3-3 does not wire the sidecar into `compose_backtest.py`.

It may add read/query helpers that future work can use, but the backtest output
must remain unchanged until a later slice designs calibration.

Future unlock stages:

| Stage | Minimum evidence | Allowed next step |
|---|---:|---|
| Observation | 30 closed days | Compare live sidecar stats with current evidence text |
| Calibration candidate | 180 closed days | Design v0.4 derivatives-aware backtest |
| AT research candidate | 365 closed days + stable ops | Offline AT feature analysis only |
| Trading input candidate | Backtest quality improvement is demonstrated | Human-approved AT integration design |

This keeps the doctrine clear: sidecar existence alone does not lift the AI Auto
Trader gate.

## Future Extensions

The schema should allow later additions without changing the v0.3-3 purpose:

- global long/short account ratio daily history;
- top-trader long/short position ratio daily history;
- taker buy/sell volume;
- external historical OI/funding provider;
- true liquidation stream/provider data;
- sidecar-derived derivative score and backtest calibration.

Do not add those in v0.3-3 unless implementation discovery shows that storing a
future-compatible empty section materially reduces later churn.

## Testing

Required focused tests:

- OI daily aggregation:
  - groups 4h samples by UTC day;
  - drops current UTC day;
  - computes sample count, completeness, first/last/avg, and growth.
- Funding daily aggregation:
  - groups funding events by UTC day;
  - drops current UTC day;
  - computes signed and absolute aggregates.
- Existing sidecar merge:
  - upserts duplicate `(asset, bucket_start)`;
  - preserves older history;
  - sorts ascending;
  - prunes retention.
- Missing existing sidecar:
  - bootstraps from provider windows.
- Sidecar failure:
  - public assets/backtest path remains safe;
  - previous sidecar is preserved.
- Producer dry-run:
  - writes temp sidecar;
  - validates public zod snapshots;
  - removes all temp files.
- Ops auto-commit scope:
  - includes exactly the three intended live data files;
  - does not sweep unrelated staged changes.

## Rollback

Rollback must be simple:

1. Revert the implementation commit.
2. Leave `data/pivot_derivatives_history.live.json` in git history as a data
   artifact, or remove it in a separate explicit data cleanup commit.
3. Existing public `pivot_assets.live.json` and `pivot_backtest.live.json`
   continue to work.
4. AI Auto Trader gate remains closed.

## Acceptance Criteria

v0.3-3 implementation is acceptable when:

- `data/pivot_derivatives_history.live.json` is produced by dry-run and live
  paths.
- Sidecar history contains only closed UTC-day buckets.
- Existing public assets/backtest JSON schemas remain unchanged.
- Existing Turning Point UI/API routes remain unchanged.
- No new secret is required.
- No new scheduler is required.
- The run_daily auto-commit path explicitly scopes all committed files.
- Sidecar failure preserves public snapshot generation and previous sidecar
  data.
- Python tests pass.
- `producer.run_producer --dry-run` exits 0.
- Existing zod snapshot validation passes.
- The design still states that AI Auto Trader integration is gated on later
  calibration, not merely on sidecar existence.
